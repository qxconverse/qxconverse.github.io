数据库相关

发表于 2019-07-31 | 更新于: 2021-03-02 | 分类于 代码 | 浏览 次
字数统计: 1.9k | 阅读时长 ≈ 6

很多数据类型的选择其实是没有黑白之分的，在不同的场景下针对一个数据可能需要不同的数据类型。

Collation的概念是字符比较的规则，每种字符集都会有其默认的Collation。

MySQL中有个存储引擎的概念。主要分为InnoDB和MyISAM。前者相较于后者，其区别主要在于它支持事务、外键和行级锁（命中主键的情况下）。

为什么要加索引，其目的是干嘛的？索引的本质是一种数据结构。最直接的类比对象是字典。索引会将某列的数据放在数据结构中，并且保存指向某行的地址值。基本原则是只如果表中某列在查询过程中使用的非常频繁，那就在该列上创建索引。

数据库中设置非空，最好设置默认值；且insert语句中涉及到的字段，不能传入null，要么insert语句不加该字段。也就是说如果可能为空，就要加判断，为空时不插入；update很简单，非空就更新。

Mybatis中的注意点

mybatis中#和$的区别

#会进行预编译，按照类型匹配的，用于变量替换。

$不进行数据类型匹配，简单的字符串拼接，一般不能用占位符的用它

如果两个都可以用，最好用#

mybatis中判断不为空，且不为空字符串

mybatis中判断不为空，且不为空字符串，适合字符串的变量；如果是整数型变量，当整数不为0时，没问题；但是如果整数为0，那么会判断它的值为空字符串，导致where中不起作用。

mybatis查询没有数据时

mybatis查询没有数据时返回的list是空集合，即size=0 并不是null。

mybatis如果只有一个参数

mybatis如果只有一个参数，并且这个参数在sql中进行了类似<if test="param != null "></if>的判断，需要加@Param。否则会都匹配进去

连接池概念

Druid是为监控而生的数据库连接池

和Java里面线程池、httpclient里面的http请求池类似，凡是涉及到连接、断开操作的东西，都可以有池的概念。为的是更好地实现资源复用。数据库连接池的基本原理是在内部对象池中维护一定数量的数据库连接，并对外暴露数据库连接获取和返回方法。连接池中包含的信息包括各种连接数的值（当前值、峰值、总值等）。

发现很多地方都有idle这个字眼，idle的意思是：懒散的、闲置的。对于连接池的细节，其为一个生产者/消费者模式，使用ReentrantLock + Condition的方式，即锁与同步方式

Condition是为了代替传统的Object的wait()/notify()方法，实现线程间的协作。其核心函数为await()、signal()方法，作用分别对应wait()和notify()。

await：释放锁、沉睡、等待唤醒；signal：获取锁，开始做事，完毕后，调用signal方法，唤醒沉睡的线程。

数据库事务的4种隔离级别

未授权读取（read uncommitted）

授权读取（read committed）

可重复读取（repeatable read）

序列化（serializable）

为什么会提出这四个概念，是因为在一个事务执行的过程中，可能出现以下几种情况：

  1. 更新丢失：两个事务同时更新一行数据，一个事务对数据的更新把另外一个事务对数据的更新覆盖了。不能应对并发
  2. 脏读：一个事务读取到了另外一个事务未提交的数据操作结果
  3. 不可重复读：一个事务对同一行数据重复读取2次，但是得到了不同的结果
  4. 虚读：事务T1读取了某一数据后，事务T2对其做了修改，当事务T1再次读该数据时得到了与第一次不同的值
  5. 幻读：事务在操作的过程中进行2次查询，第一次查询到的结果包含了第一次查询中未出现的数据或者缺少了第一次查询中出现的数据。这是因为在两次查询过程中另外一个事务插入数据造成的

未授权读取（我在写，你就不能写；但是你可以读）就是一个事务可以读取另一个未提交事务的数据。
如果一个事务已经开始写数据，则另外一个事务不允许同时进行写操作，但是允许其它事务读此行数据。该隔离级别可以通过“排他写锁”实现。
避免了更新丢失，却可能出现脏读。也就是说事务B读取到了事务A未提交的数据

授权读取（我在写，如果还没有提交，你不能读）
就是一个事务要等另一个事务提交后才能读取数据。读取数据的事务允许其它事务继续访问该行数据，但是未提交的写事务将会禁止其它事务访问该行。
避免了脏读，但是可能出现不可重复读。事务A事先读取了数据，事务B紧接着更新了数据，并提交了事务，而事务A再次读取该数据时，数据已经发生了变化。

可重复读取（我在读，你不能写）
就是在开始读取数据（事务开启）时，不再允许修改操作读取数据的事务将会禁止写事务（但允许读事务），写事务则禁止任何其它的事务。避免了不可重复读和脏读，但是有时可能出现幻读。可以通过“共享读锁”和“排他写锁”实现。
重复读可以解决不可重复读问题。不可重复读对应的是修改，即UPDATE操作。但是可能还会有幻读问题。因为幻读问题对应的是插入INSERT操作，而不是UPDATE操作。

序列化，事务只能一个接着一个地执行，但不能并发执行。可以解决幻读问题

总结：对于大多数应用程序，可以优先考虑把数据库的隔离级别设置为read committed。可以避免脏读，而且有较好的并发性能。尽管可能导致不可重复读、幻读和第二类丢失更新这些并发问题。可以由应用程序采用悲观锁或乐观锁来控制。MySQL默认的隔离级别就是repeatable read。
