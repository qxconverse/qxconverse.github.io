数据库事务四种隔离级别

发表于 2018-09-02 | 分类于 代码
字数统计: 878 | 阅读时长 ≈ 2

未授权读取（read uncommitted）

授权读取（read committed）

可重复读取（repeatable read）

序列化（serializable）

为什么会提出这四个概念，是因为在一个事务执行的过程中，可能出现以下几种情况：

  1. 更新丢失：两个事务同时更新一行数据，一个事务对数据的更新把另外一个事务对数据的更新覆盖了。不能应对并发
  2. 脏读：一个事务读取到了另外一个事务未提交的数据操作结果
  3. 不可重复读：一个事务对同一行数据重复读取2次，但是得到了不同的结果
  4. 虚读：事务T1读取了某一数据后，事务T2对其做了修改，当事务T1再次读该数据时得到了与第一次不同的值
  5. 幻读：事务在操作的过程中进行2次查询，第一次查询到的结果包含了第一次查询中未出现的数据或者缺少了第一次查询中出现的数据。这是因为在两次查询过程中另外一个事务插入数据造成的

未授权读取（我在写，你就不能写，但是你可以读）就是一个事务可以读取另一个未提交事务的数据。
如果一个事务已经开始写数据，则另外一个事务不允许同时进行写操作，但是允许其它事务读此行数据。该隔离级别可以通过“排他写锁”实现。
避免了更新丢失，却可能出现脏读。也就是说事务B读取到了事务A未提交的数据

授权读取（我在读，你可以读；但是我在写，如果还没有提交，你不能读）
就是一个事务要等另一个事务提交后才能读取数据。读取数据的事务允许其它事务继续访问该行数据，但是未提交的写事务将会禁止其它事务访问该行。
避免了脏读，但是可能出现不可重复读。事务A事先读取了数据，事务B紧接着更新了数据，并提交了事务，而事务A再次读取该数据时，数据已经发生了变化。

可重复读取（我在读，你不能写，可以读）
就是在开始读取数据（事务开启）时，不再允许修改操作读取数据的事务将会禁止写事务（但允许读事务），写事务则禁止任何其它的事务。避免了不可重复读和脏读，但是有时可能出现幻读。可以通过“共享读锁”和“排他写锁”实现。
重复读可以解决不可重复读问题。不可重复读对应的是修改，即UPDATE操作。但是可能还会有幻读问题。因为幻读问题对应的是插入INSERT操作，而不是UPDATE操作。

序列化，事务只能一个接着一个地执行，但不能并发执行。可以解决幻读问题

总结：对于大多数应用程序，可以优先考虑把数据库的隔离级别设置为read committed。可以避免脏读，而且有较好的并发性能。尽管可能导致不可重复读、幻读和第二类丢失更新这些并发问题。可以由应用程序采用悲观锁或乐观锁来控制。MySQL默认的隔离级别就是repeatable read。

计算机思维

发表于 2018-08-26 | 分类于 代码
字数统计: 620 | 阅读时长 ≈ 2

在计算机中，有如下的思维，一定要熟记于心，无论遇到什么问题，都可以往这方面思考，正所谓万精油！！

  1. 缓存思想：缓存思想特别重要，这是很多设计中必出现的，无论在操作系统层面还是具体应用层面。比如在内存和CPU之间有寄存器；比如在Redis中有缓存机制；Mybatis也有缓存机制，当某些记录很大，而且基本上不会变，且查询频率很高，那么利用缓存是明智的选择。消息队列
  2. Hash+链表、树形结构思想：说到hash一般会想到hashtable，hashmap等等，树形结构就更多了。共同点是，说到查找，它们是高频词汇。对于无序的东西，我们的查找时间复杂度是n级别的。所以为了优化，会采用2种措施，一种是建立索引，即hash表，计算key值，然后放到对应的位置，如果出现碰撞，通常是加个链表。这样复杂度可以控制在1级别的。凡事都有优缺点，虽然查询快了，但是它是无序的。而如果采用第二种措施，即用树形结构，比如二叉树（红黑树）、B+树等。基本是可以实现logn级别的，虽然慢一些，但其优点是有序。凡是涉及到查找，要想到它们！在ElasticSearch中有这些思想的存在（倒排索引，基于单词文档矩阵模型）。
  3. 分布式思想：这个不用说了，当应用大了以后，必然出现。涉及到分布式，那么一定会涉及到同步、一致性等等问题。
  4. 解耦思想：这个思想很重要，无论在什么地方都会体现，任何系统，都应当考虑到解耦。
  5. top-down思想：任何时候可以先从宏观整体考虑，然后再到具体的部分。递归是一种体现。
  6. 主从思想：这一思想常常出现在分布式中
  7. DP思想：这个不用说，算法中最难以理解，也是最能解决复杂问题的一种思想。
  8. 双倍扩容思想：对于一些动态容器，比如C++中的vector扩容，java中的hashmap扩容，Redis中的dict的rehash过程。多多少少有基于原来容量2倍来进行扩容的思想。
  9. 拦截过滤思想
  10. 注册通知（发布、通知）思想，反应器模式

zookeeper知识

字数统计: 906 | 阅读时长 ≈ 3

zk是一个分布式协调服务。

对比在进程内的协调、同步机制等等，而zk是基于分布式系统的，说白了就是有很多个机器。每一个机器类比之前进程里面的每一个线程。分布式显然问题更多一些，最主要的就是网络的不可靠性。

zk可以做的事情包括：对配置信息的管理，名字服务，提供分布式同步，集群管理。

  1. 配置管理：因为涉及到多台机器，每台机器都要有这样的配置，而且配置文件比较多。这时就可以使用zk服务，它使用Zab这种一致性协议来提供一致性。
  2. 名字服务：提供统一的入口来维护各个服务的名字。
  3. 分布式锁：某个时刻只让一台机器去干活，当这台机器出现问题时，就释放锁，给其它机器（服务）。这种设计思想很普遍，在HBase、Redis（选举新的master）中都有涉及，叫做LeaderElection（leader选举）。
  4. 集群管理：集群中的节点要对其内部的变化做出相应的对策。而且当消费者访问某个服务时，就要有服务发现的机制。Dubbo就是采用zk作为服务发现的底层实现。Kafka消息队列也是采用zk作为Consumer的上下线管理。

zk的启动过程：首先会解析配置文件，接着会初始化ServerCnxnFactory，这个是用来接收客户端连接的，说白了就是一个Tcp Server。zk中有2种实现tcp server的方法，一个是用Java原生的NIO方式，一个是使用Netty。多路复用的IO模型，一个典型的Reactor模型。创建几个SelectorThread处理具体的数据读取和写出。然后就是创建一个AcceptThread线程接收客户端的连接。后面是主要的过程

  1. 初始化FileTxnSnapLog
  2. 初始化ZKDatabase
  3. 初始化决议validator
  4. leader选举，这一步非常重要也是zk里面最精华的部分（FastLeaderElection方式，每个集群中的节点都有一个状态looking、following、observing、leading，具体的选举过程可以参考:https://www.cnblogs.com/yuyijq/p/4116365.html）

zookeeper的核心算法是Raft协议，主要用于模拟集群，其主要有3种角色：

  1. Leader
  2. Follower
  3. Candidate

其实整个的选举过程类似于狼人杀中的选举警长。当然这个要更加复杂些。有一个很重要的概念叫做任期（Term）

今天终于理解了心跳信息是个什么东西，原来当选举完成后，leader会定期给所有的follower发送心跳信息维持其统治，如果follower一段时间没有收到心跳，那么可以认为leader已经挂掉了。。。这时就会重新选举，任期加一。

Redis的集群数据库的设计原理很它很相似，说白了。这些是共通的，要联系起来记忆消化。

当客户端在发送数据给leader后，这时数据属于未提交的状态。首先leader会把数据发给所有的follower等待它们的确认，一旦有超过半数节点给出确认接收的消息后，leader再给客户端发送确认消息，接着再给每个follower节点发送消息表示该数据已经提交了。整个过程很依赖于一致性。即保证所有节点的数据的一致性。

分布式架构的消息通信方式，主要有基于RPC的和基于消息队列的，还有基于一般的Http请求的。

关于5种I/O模型的理解

以喝咖啡举例， 在网上点了咖啡粉外卖，要等外卖小哥送咖啡，然后泡咖啡，喝咖啡

  1. 阻塞式I/O：一直站在家里等咖啡（其它什么事不做）送到，然后泡咖啡
  2. 非阻塞式I/O：不停地给外卖小哥打电话（其它什么事不做），送到后泡咖啡
  3. IO多路复用：一开始和小哥说好，到了后打电话，同时站在家里等（于此同时，他还在等其它的外卖，也是什么事不做），一旦咖啡到了，就泡咖啡
  4. 信号驱动式IO：干自己的事情，外卖小哥到了后打电话让你取咖啡，然后你泡咖啡
  5. 异步IO：一直干自己的事情，整个事情都让女朋友来做，做好了通知自己过去喝

前面3种提到的IO都是同步的，与之相对应的是异步IO。这是一个维度，还有一个是阻塞的维度。传统的是阻塞的IO，不管是C端还是S端，都是阻塞的。接着是非阻塞的IO，主要通过C端的轮询来实现。后面的多路复用，它也是非阻塞的，通过选择器和通道来实现。主要采用的是反应器（Reactor）模式。涉及到事件的注册和订阅。

很早很早之前，Java的api只支持由本地系统套接字库提供的阻塞函数。阻塞体现在服务器端的监听过程，直到有新的连接，这时就建立一个双方通信的socket。接着它继续监听。其缺点是只能同时处理一个连接，要管理多个并发的客户端连接，需要开启多个线程。

Java中的单例模式

发表于 2018-08-17 | 分类于 代码
字数统计: 157 | 阅读时长 ≈ 1

参考博客：http://cantellow.iteye.com/blog/838473

单例模式的优缺点及使用场景：http://www.cnblogs.com/damsoft/p/6105122.html

优点：

  1. 避免对共享资源的多重占用
  2. 节约系统资源
  3. 对唯一实例的受控访问
  4. 可以控制实例的数目

缺点：

  1. 职责过重，违反了“单一职责原则”
  2. 单例模式中没有抽象层，不方便扩展
  3. 可能会导致内存溢出
  4. 周期过长，可能被垃圾回收

一般采用饿汉式，若对资源十分在意可以采用静态内部类，不建议采用懒汉式及双重检测

Java抽象类和接口的区别

发表于 2018-08-17 | 分类于 代码
字数统计: 251 | 阅读时长 ≈ 1

两者的共同点：

  1. 都可以实现抽象化，都不能实例化
  2. interface的实现类以及abstract class的子类都需要实现已经声明的抽象方法

区别：

  1. 一个类可以实现多个interface，而只能继承一个抽象类
  2. interface强调特定功能的实现，而abstract class强调所属关系，所以interface设计思想是like-a，而abstract class的设计思想是is-a
  3. interface不能有自己的私有变量和方法，而abstract class可以有
  4. interface里面的所有方法都是抽象方法，实现类必须实现，而abstract class的子类可以有选择地实现。具体是，abstract class中的类不都是抽象方法，抽象方法子类必须实现，而非抽象方法，它自己实现。而且对于抽象方法，子类可以选择继续让它抽象，由该子类的子类去实现，这是该子类必须也声明为abstract class。

Java虚拟机

发表于 2018-08-17 | 分类于 代码
字数统计: 943 | 阅读时长 ≈ 3

java内存模型：
java虚拟机是一个虚拟的环境，占据着一部分主机的内存。也就是说这是一个中间桥梁，java在运行时不是在赤裸裸的内存上进行的，而是在虚拟机上。

这个区域主要分为2大类。一个是线程隔离的，一个是线程共享的。而它们的生命周期也与此有关。一类是随着虚拟机进程的，而另一类是随着线程的。也就是说线程没了，它也就没了。

一共有5个区域，其实记忆的时候可以类比C/C++中的堆和栈。好！那么这5个区域分别为：

  1. 程序计数区（线程隔离）
  2. Java虚拟机栈（栈，线程隔离）
  3. 本地方法栈（栈，线程隔离）
  4. Java堆（堆，线程共享）
  5. 方法区（堆，线程共享）

但凡涉及到内存，首先就要想到堆和栈。毕竟从本科的时候起，老师们就开始一直灌输这些了。什么栈溢出，堆溢出。其实，说白了，就是递归的深度超过了一定的数目，和内存不够分配了。一个说的是纵向，一个是横线。类比于算法中的广度优先与深度优先，深度优先搜索使用的一定是栈，因为涉及到递归；而广度优先搜索使用到的是队列，它涉及到堆。

仔细思考一下，在C/C++中，是否记得一般的运行时局部变量是存放在栈中的，而程序员申请的空间是存放在堆中的。堆位于地址较高的地方，而栈位于较低的地方。一样的啊！！！Java也是这个套路，虚拟机栈和本地方法栈就是为此服务的，只不过一个给Java方法服务，而另外一个给本地方法（Native）服务！

其实宏观地可以结合算法和数据结构以及计算机体系来理解这些，如下：

计算机体系 算法 数据结构 计算机思想
栈区（内存中连续） 深度优先搜索 栈（stack） 递归
堆区（内存中非连续） 广度优先搜索 堆（heap），优先队列（基于堆） —-

看到了吧，其实并没有想象中的那么难以记忆，关键在于总结和联想！！

再来看看垃圾回收机制吧！

垃圾回收说白了就是针对堆的，仔细想想，java中所有的对象都是存放在堆中的，所以这个地方是最需要进行垃圾回收的。这个时候就要问自己几个问题，第一，什么时候要回收？第二，怎么回收？
关于垃圾回收的方法主要有3种：

  1. 标记-清理（先标记，然后清理）缺点是容易造成碎片
  2. 复制（分为2个区域，每次把存活的对象移动到另外一个区，然后一次性清除当前区的对象）
  3. 标记-整理（先标记，然后整理，也就是说把存活的移动到区域的某一端，然后清理掉端外面的即可）

目前最流行的当然是综合的方法，新生代中的对象几乎都很短命，复制成本低，所以用方法2；而老年代中放的是长命的对象，往往比较大，所以用方法1或3。

好了，有了垃圾回收的方法，那么就要问了，对象放在上面地方呢？有以下3个策略。

  1. 对象优先放在Eden区
  2. 大对象放在老年代
  3. 长期存活的对象将进入老年代（注意这里有个对象年龄的概念，年龄达到一定程度即可进入老年代）
