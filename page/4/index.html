关于5种I/O模型的理解

以喝咖啡举例， 在网上点了咖啡粉外卖，要等外卖小哥送咖啡，然后泡咖啡，喝咖啡

  1. 阻塞式I/O：一直站在家里等咖啡（其它什么事不做）送到，然后泡咖啡
  2. 非阻塞式I/O：不停地给外卖小哥打电话（其它什么事不做），送到后泡咖啡
  3. IO多路复用：一开始和小哥说好，到了后打电话，同时站在家里等（于此同时，他还在等其它的外卖，也是什么事不做），一旦咖啡到了，就泡咖啡
  4. 信号驱动式IO：干自己的事情，外卖小哥到了后打电话让你取咖啡，然后你泡咖啡
  5. 异步IO：一直干自己的事情，整个事情都让女朋友来做，做好了通知自己过去喝

Git的基础知识

git的工作区就是保存在电脑上的实体文件夹，而stage称为暂存区，add时添加到里面，master就是主分支，而HEAD可以理解为一个指针，指向当前的版本。

先手动删除文件，然后使用git rm <file>和git add <file>效果是一样的。

切换到原来的分支后，文件的内容并没有变化，因为我们是在另外的一个分支上开发的。

Git鼓励大量使用分支：

查看分支：git branch

创建分支：git branch <name>

切换分支：git checkout <name>

创建+切换分支：git checkout -b <name>

合并某分支到当前分支：git merge <name>

删除分支：git branch -d <name>

合并分支时，加上--no-ff参数就可以用普通模式合并，合并后的历史有分支，能看出来曾经做过合并，而fast forward合并就看不出来曾经做过合并。

新建分支的时候，一定要注意是在哪个分支上建立的。

在实际开发中，我们应该按照几个基本原则进行分支管理：

首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；我们每个人都有自己的分支，时不时地往dev分支上合并就可以了。

所以，团队合作的分支看起来就像这样：

当修改完bug分支后，要及时地在master上删除该分支，如果切换到dev分支上再删除bug分支，就会报错。

通常涉及到多个人开发，比如A和B。B从github上clone了项目。并且新建并切换到dev分支了。

这个时候A（你）和B（你的小伙伴）对同一个文件进行了修改，发生了冲突。假设B先push了，而你后面想push，就会发生错误。所以可以先pull一下。pull也可能失败，原因是没有指定本地dev分支与远程origin/dev分支的链接，根据提示，设置dev和origin/dev的链接：
git branch --set-upstream-to=origin/dev dev

这回pull成功，但是合并有冲突，需要手动解决，解决的方法和分支管理中的解决冲突完全一样。解决后，add，ci,再push。这样就完美解决了冲突。

这时B如果要再次编辑，首先需要pull一下。然后再进行后续操作。如果他不pull，push后有冲突，那么解决方法同上A所做的。

git中对比2个版本之间的改动，比如下面的例子

git diff xxx1 xxx2 --stat

在add之前，查看修改了哪些代码，很实用。

修改了冲突后，记得add，commit，pull然后再push

在pull的时候，如果本地有修改过的，那么可以强制恢复到上一次提交的位置。

修改了代码后，想去掉修改：git co.

修改了代码后，add了，但想撤回：git reset HEAD .

修改了代码后，add/commit了，但想撤回：git reset —hard xxxxxxxxx

上传文件到linux的命令：rz

解决git pull/push每次都需要输入密码问题：https://blog.csdn.net/nongweiyilady/article/details/77772602

Git最佳实践：

  1. 要经常提交（细粒度地提交）
  2. 将功能分成很多小块，每次全部完成某个小块时再提交，不要还没完成就提交
  3. 如果没完成，而需要切换分支或pull操作，可以先stash一下
  4. 提交（commit）之前要确保已有的代码测试通过，且不影响到其它代码
  5. 提交message要说明，为何变更以及和之前的区别，用现在时态
  6. 版本控制与备份有区别，应当遵循前面5条实践规则，而不要仅仅把它当做备份的工具
  7. 要经常使用branch，比如开发的新特性分支、修改的bug分支等等
  8. 工作流程的规则要保持一致，至少要和团队一致

Pick a starting point.
