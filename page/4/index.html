代码整洁之道总结

注释

删除掉注释掉的代码

代码中充斥着大量旧代码注释会造成额外的维护负担。谁都不记得这些注释的作用了，又不知道能否直接删除

删除掉冗余注释

注释只应该描述有关代码和设计的技术性信息；如果注释描述的是某种充分自我描述了的东西，那么注释就是多余的

类

类的三大原则

  1. 单一职责原则（SRP）：一个类应当只有一个职责
  2. 开放闭合原则（OCP）：对扩展开发、对修改闭合，尽量保证在不修改原有代码的情况下，对软件进行扩展
  3. 依赖倒置原则（DIP）：用抽象构建框架，用实现扩展细节

SRP原则的解释

OCP/DIP原则解释

函数

不要有过多的参数

函数参数越少越好，如无必要不要使用三个以上的参数函数；从测试的角度来看，如果参数过多，那么要覆盖所有可能值的组合让人生畏。

删除掉不被调用的方法

不被调用的方法放在代码中可能会误导开发者，增加代码的阅读复杂度

不要用标识参数，比如传入的布尔值变量

  1. 阅读困难，本质上相当于无意义的变量
  2. 可能导致了一个函数并没有只做一件事
  3. 向函数传入布尔值用于区分不同业务的做法很丑陋，应该拆分为多个函数

用枚举类实例代替

拆分为多个函数

一般性问题

避免重复，多抽象

  1. 重复不断的代码，可以封装成一个方法
  2. 在不同模块中重复出现的switch/case、if/else链，可以用多态替代之
  3. 采用类似算法但具体代码行不同的模块，可以用模板方法模式或策略模式修正

使用异常而非错误码

异常机制可以让正常流程代码与错误代码互相分离，不耦合。如果全部由错误码来做，阅读复杂性会提高

使用异常来代替

在最外层（比如Controller层），二者可以结合起来，捕捉到异常后，再返回给前端统一的错误码。

返回空的数组或集合不要返回null

对于数组或是集合类型的变量，返回空要比返回null更安全些，如果为null，在不进行校验的情况下很容易出现空指针问题

其它

  1. 隔离的实现方式可以是：IoC/DI
  2. 扩容的实现方式可以是：切面（aop编程）、Java代理
  3. 较易的测试，会间接导向保持类短小且目的单一、符合SRP/DIP的设计方案

字段 示例 详解
Request URL xxx 请求的URL
Request Method GET 请求的方法
Status Code 200 状态码
Remote Address xxx 远程地址（服务器地址）
Referrer Policy no-referrer-when-downgrade Referrer策略，可参考博客
- Response Headers
Content-Type application/json;charset=UTF-8 响应的内容类型以及编码
Date Tue, 30 Jul 2019 10:30:54 GMT 响应的时间
Transfer-Encoding chunked 传输编码形式，chunked表示数据以一系列分块的形式进行发送
- Request Headers
Host xxx 服务器的域名
Connection keep-alive 决定当前的事务完成后，是否会关闭网络连接。keep-alive表示网络连接是持久的，不会关闭，对同一个服务器的请求可以继续在该连接上完成。
Pragma no-cache
Cache-Control no-cache 缓存机制
X-Requested-With XMLHttpRequest XMLHttpRequest表示为Ajax请求（异步）
User-Agent xxx 识别发起请求的用户代理软件的应用类型、操作系统、软件开发商以及版本号
Content-Type application/json 用于指示资源的MIME类型
Accept */* 告知客户端可以处理的内容类型
Referer xxx 当前请求页面的来源页面的地址
Accept-Encoding gzip, deflate 客户端能够理解的内容编码方式
Accept-Language zh-CN,zh;q=0.9,fr;q=0.8,zh-TW;q=0.7,en;q=0.6 允许客户端声明它可以理解的自然语言
Cookie xxx 服务器发送到用户浏览器并保存在本地的一小块数据，它会在浏览器下次向同一服务器再发起请求时被携带并发送到服务器上。

Access-Control-Allow-Origin: 对于不需具备凭证（credentials）的请求，服务器会以*作为通配符，从而允许所有域都具有访问资源的权限。

401 VS 403
401：表示的是 Unauthentiated: 需要先提供有效凭证验证身份（然而通常返回这个状态码时，响应的响应信息都是Unauthorized）
403：表示的是 Unauthorized：你已经提供的有效凭证，但你的身份不不能访问这个资源。

请求耗时中的TP值

什么是请求耗时

请求耗时指的是某个请求从发出到服务器响应花费的总时间，我们经常需要将它作为一个监控指标来衡量系统的服务质量。很显然，一个服务的请求耗时越少，表明该服务的质量越高。

如何获取请求耗时

特别地，可以考虑该服务中某个接口的请求耗时。空谈请求耗时是没有意义的，我们需要度量某个时间范围内的整体请求耗时。比如过去1分钟内的请求耗时、昨天的请求耗时等等。

有了范围，我们就要统计该范围内接口所有请求的耗时。假设过去1分钟内一共有10次请求，其耗时分别为

[34ms, 10ms, 8ms, 12ms, 20ms, 9ms, 18ms, 29ms, 3ms, 15ms]

平均值统计

思考一下，我们如何计算该接口的整体请求耗时呢？也许你会说，平均值啊，多么简单而粗暴的统计方式。于是你计算如下：(34 + 10 + 8 + 12 + 20 + 9 + 18 + 29 + 3 + 15) / 10 = 15.8ms。看上去挺准确的，差不多在15ms左右，没有问题。

平均请求耗时看起来很简单，但是却掩盖了一个重要的细节：很可能大多数请求都是很快的，但是长尾请求速度却很慢。比如，再看下面的10次请求耗时：[34ms, 500ms, 8ms, 12ms, 20ms, 9ms, 480ms, 29ms, 3ms, 15ms]

计算其平均值，结果为111ms。假设我们的报警值为120ms。虽然大多数请求在120ms内完成了，但20%的请求（这里为2个）却慢了20倍左右！此时报警不会发出，但是服务的确在长尾耗时上出现了巨大的变化。

因此，我们可以得出一个结论，请求耗时指标不应该简单地由平均值来定义。

百分位值统计

这里就引入了统计学中的一个概念，分布。我们将所有耗时从小到大排序，得到下面的耗时列表，方便起见，去掉ms：[3, 8, 9, 12, 15, 20, 29, 34, 480, 500]。

我们定义Top Percent的概念，即标题中的TP值，也叫百分位值。

TP值：指在一个时间段内，统计该接口每次调用所消耗的时间，并将这些时间按从小到大的顺序进行排序，并取出第（总次数 $\times$ 百分比）个值。

比如，在上面的例子中，TP50 = 第（10$\times$50%）个值，即TP50 = 15ms；依此类推，TP90 = 第（10$\times$90%）个值，即TP90 = 480ms。

稍微转换一下视角，就可以得到下面的结论：

  1. 50%的请求会在15ms内（小于等于）完成
  2. 90%的请求会在480ms内完成
  3. 95%的请求会在500ms内完成
  4. 99%的请求会在500ms内完成

利用百分位值可以帮助我们关注其分布性：高百分位，例如99%，体现了最差的情况，而50%则体现了普遍情况。

结论

下图中的蓝色线：某段时间范围内，接口请求的耗时由低到高排序所形成

  1. TPX，表示Top Percent X的值，通常情况下，相同的X，TPX值越大，表明接口整体的耗时越长
  2. 对于同一个接口，使用X值越大的TPX，统计到的值越大，可以看做条件越苛刻（对服务质量的要求更高）

接着来解释一下这两个结论。

先来看看同样的TP标准，其值越大，是否意味着其请求耗时要高一些，也就是服务质量要差一些。以TP90举例，看下面的图：

很明显，我们会认为接口2的请求耗时要高于接口1的请求耗时，即我们认为接口1服务质量要高些。

我们其实更倾向于关注高分位TP值，因为如果99%的系统行为都正常的话，那50%部分肯定也就是正常的。所以高分位值对服务性能有更高的要求。

比如120ms作为报警值，如果取TP50，因为15ms < 120ms，所以不会有报警；而取TP90的话，因为480ms > 120ms了，因此就会报警了。可见TP90更加苛刻一些。
