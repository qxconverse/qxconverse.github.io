监控的red原则

指标涉及到以下：

系统状况：可达性、CPU、内存、JVM等

各项服务：MySQL、Dubbo、Kafka等

接口访问：400、5xx日志数

  1. rate
  2. error
  3. duration

SLI指的是服务质量指标

常见的有：请求延迟（处理请求所消耗的时间）、错误率（请求处理失败的百分比）、系统吞吐量（每秒请求数）。还有一个可用性，代表服务可用时间的百分比。

需要注意的是：监控错误回复的延迟很重要，要和正确的分开来

错误分为多种：显示错误（比如500），隐式错误（虽然返回200，但是回复里面有错误的内容），策略原因导致的失败（比如，规定响应时间高于1s算是失败）

除了red，还有饱和度：即服务容量现在有多满，比如 CPU、内存、IO使用率等

报警出来的信息不应该需要主观判断，否则就失去了其意义（就是琐事）。报出来的警，必须是明确的、客观、可以无需思考立即执行的。

基础设施监控
  • CPU
  • 内存
  • 磁盘（存储健康）
  • 网卡（网络健康）

个人总结

关于样式等
  1. 字体要统一，可以分成几大类，比如一级标题、正文等等字体
  2. 字体大小要统一，比如一级标题、二级标题、正文、说明、图例等等几类字体大小
  3. 字体样式要统一、比如哪些地方加粗、哪些地方标什么颜色等等
  4. 字体颜色要统一、比如什么地方用黑色、什么地方用白色、淡蓝色等等
  5. 背景和前景颜色要能明显区分，浅色背景配深色前景，深色背景配浅色前景（比如形状、或字体）
  6. 每一页的文字要控制在能够让观众3秒内阅读（扫视）完的数量级别
  7. 每一页只讲一样东西，配以图片等
  8. 对于重点的词句，应当加粗显示、或换一个较大的字体大小
  9. 多用各种图来表示列表
  10. 当要表示某种组成关系的时候，画框架图
  11. 当要表示某种次序关系的时候，画时序图或流程图

Http头信息

字段 示例 详解
Request URL xxx 请求的URL
Request Method GET 请求的方法
Status Code 200 状态码
Remote Address xxx 远程地址（服务器地址）
Referrer Policy no-referrer-when-downgrade Referrer策略，可参考博客
- Response Headers
Content-Type application/json;charset=UTF-8 响应的内容类型以及编码
Date Tue, 30 Jul 2019 10:30:54 GMT 响应的时间
Transfer-Encoding chunked 传输编码形式，chunked表示数据以一系列分块的形式进行发送
- Request Headers
Host xxx 服务器的域名
Connection keep-alive 决定当前的事务完成后，是否会关闭网络连接。keep-alive表示网络连接是持久的，不会关闭，对同一个服务器的请求可以继续在该连接上完成
Pragma no-cache
Cache-Control no-cache 缓存机制
X-Requested-With XMLHttpRequest XMLHttpRequest表示为Ajax请求（异步）
User-Agent xxx 识别发起请求的用户代理软件的应用类型、操作系统、软件开发商以及版本号
Content-Type application/json 用于指示资源的MIME类型
Accept */* 告知客户端可以处理的内容类型
Referer xxx 当前请求页面的来源页面的地址
Accept-Encoding gzip, deflate 客户端能够理解的内容编码方式
Accept-Language zh-CN,zh;q=0.9,fr;q=0.8,zh-TW;q=0.7,en;q=0.6 允许客户端声明它可以理解的自然语言
Cookie xxx 服务器发送到用户浏览器并保存在本地的一小块数据，它会在浏览器下次向同一服务器再发起请求时被携带并发送到服务器上

Access-Control-Allow-Origin: 对于不需具备凭证（credentials）的请求，服务器会以*作为通配符，从而允许所有域都具有访问资源的权限

状态码

状态码 含义
401 表示的是 Unauthentiated: 需要先提供有效凭证验证身份（然而通常返回这个状态码时，响应的响应信息都是Unauthorized）
403 表示的是 Unauthorized：你已经提供的有效凭证，但你的身份不不能访问这个资源
