关于社交网络的感悟

发表于 2014-03-22 | 更新于: 2021-01-03 | 分类于 随想
字数统计: 1.3k | 阅读时长 ≈ 4

这是一个人人渴望在网络世界中获得存在感的时代！每天，我们都扮演着许许多多的共享与被共享的角色！这些都是我们习以为常，感觉微不足道的事件。我们每天都在不自觉地接受他人的动态和发布我们自己的动态，通过网络社交平台（最典型的是QQ空间）。空间就是我们驻扎在网络虚拟世界中的家，这是一个用大数据堆砌而成的并以时间为轴线的时空。我们每天都在经营着它的建设。数据与时间就是我们的付出，人气与存在感便是我们的收获。这样，我们就不难理解我们日常的“强迫症”了：早上起来第一件事就是登上手机QQ，接着排队买早餐、上课、课间休息、吃饭、睡觉前等随时随地，只要是无聊之时就会刷空间，打发无聊。其内容要么是见闻所感，要么是自嘲人艰，要么是吐槽社会，要么是爆料嗅事，要么是喜大普奔，要么是炫耀生活，要么是转载幽默，要么是感慨人生。。。。。。总之就是感情大杂烩！

看来（可能有点武断），手机、电脑都是这个时代极具威力的“杀猪刀”，如果运用不当！它们在无声无息中带走我们的青春。手机成了白天的朋友，电脑成了夜晚的伙伴，于是床铺成了清晨的伴侣。。。。。。

手机、电脑最大的诱惑之一乃是社交平台，现实中我们恐惧与他人交往，因此只能通过虚拟的方式去与Ta们沟通。这并非是一件坏事（毕竟加快了沟通的效率），但是这一切却导致了我们这代人的胆怯与萎靡！

我们每天渴望着嗅事、笑话，渴望着别人的赞与评论，我们更惊喜于神回复（正所谓语不惊人死不休）！也许这是一种积极乐观的心态，可是稍加分析，便可发现这是掩饰与逃避。要知道，这一切的素材都来源于现实世界，加工却是通过网络，网络是一个宽松自由的工具，可以完成我们在现实中不能完成的事情，我们追求网络中的幽默来减压，这来自现实生活的压力，而不是在现实中减压。说起来，网络只是一个桥梁，不是完全与现实不搭界的东西，这在另一方面又能反映现代社会的高压力环境了。反过来，我们又常常将网络上的表现运用到现实生活中。。。。。。

这是一个用“吐槽妙语”来渲染感情的时代，我们日常交际用语中表达情感的仅仅局限于那些少个别的词或句子。这些词句诞生于网络，传播于网络，可又来源于现实，而后又在现实中传播！这些词语太多太多，可以反映了一个时代的特征了。比如，“我和我的小伙伴们都惊呆了！”，“土豪，我们做朋友吧？”，“累觉不爱”，“人艰不拆”，“放弃治疗”，“不明觉厉”，“待我长发及腰”，“神马都是浮云”，“正能量”，“逗比”，“基友”，“貌似”，“话说”“操”等等，只要你稍微关注网络，特别是社交、论坛这一块的话，你会知道很多内涵的！似乎我们所有感情都可以用那些词句表达出来。不过，话说回来，那些词也够高度概括的了，就好似C++里面的类，经过继承与派生，都能成就你语言表达的多态性。还有一些神回复，你会发现它们是多么地全能完美！

可是，长期下去，我们的词汇量越来越少，沟通能力也会逐日下降，往往是一个群体，一代人同时受害的。因为任何一个个体倘若不以那样的方式沟通，就会被其他人所排斥。于是，我们发现，这是一个集体“掉节操”的时代，也就并不稀奇了！

随着时代的发展，人类越来越懒，很多事都依赖于人工智能。电脑以及其中的多数软件便能提供这些。于是在网络社交平台上，我们的“程序猿”与一些网络大神已经为我们设计了很多固定的话语与表情来表达一连串的感情。就比如QQ里面的一个“（抠鼻）”就能表达巨量的感情。于是，我们常常用表情来交流，这样就显得委婉、贴切、卖萌多了！可是苦闷的事情也来了，倘若跟异性交流，如果对于你的很长的、正经肺腑的话语，她总是以一个表情来打发的话，那么，你就是一个悲剧的人了。

本文仅仅针对于网络社交平台而做的浅谈，不针对其它的。更没有针对某些人或是群体，毕竟我的生活也离不开这些。。。。。

Http头信息

字段 示例 详解
Request URL xxx 请求的URL
Request Method GET 请求的方法
Status Code 200 状态码
Remote Address xxx 远程地址（服务器地址）
Referrer Policy no-referrer-when-downgrade Referrer策略，可参考博客
- Response Headers
Content-Type application/json;charset=UTF-8 响应的内容类型以及编码
Date Tue, 30 Jul 2019 10:30:54 GMT 响应的时间
Transfer-Encoding chunked 传输编码形式，chunked表示数据以一系列分块的形式进行发送
- Request Headers
Host xxx 服务器的域名
Connection keep-alive 决定当前的事务完成后，是否会关闭网络连接。keep-alive表示网络连接是持久的，不会关闭，对同一个服务器的请求可以继续在该连接上完成
Pragma no-cache
Cache-Control no-cache 缓存机制
X-Requested-With XMLHttpRequest XMLHttpRequest表示为Ajax请求（异步）
User-Agent xxx 识别发起请求的用户代理软件的应用类型、操作系统、软件开发商以及版本号
Content-Type application/json 用于指示资源的MIME类型
Accept */* 告知客户端可以处理的内容类型
Referer xxx 当前请求页面的来源页面的地址
Accept-Encoding gzip, deflate 客户端能够理解的内容编码方式
Accept-Language zh-CN,zh;q=0.9,fr;q=0.8,zh-TW;q=0.7,en;q=0.6 允许客户端声明它可以理解的自然语言
Cookie xxx 服务器发送到用户浏览器并保存在本地的一小块数据，它会在浏览器下次向同一服务器再发起请求时被携带并发送到服务器上

Access-Control-Allow-Origin: 对于不需具备凭证（credentials）的请求，服务器会以*作为通配符，从而允许所有域都具有访问资源的权限

状态码

状态码 含义
401 表示的是 Unauthentiated: 需要先提供有效凭证验证身份（然而通常返回这个状态码时，响应的响应信息都是Unauthorized）
403 表示的是 Unauthorized：你已经提供的有效凭证，但你的身份不不能访问这个资源

  8. 醋
  9. 蒜末

  11. 花生米

蒜蓉蒸茄子

食材
  1. 茄子
  2. 大蒜
  3. 玉米油
  4. 盐
  5. 六月鲜酱油
做法
  1. 大蒜切碎成蒜蓉
  2. 茄子切半，并划几道形成裂口
  3. 在茄子上均匀抹上蒜蓉、油、盐
  4. 将茄子放入电饭锅蒸，并加入六月鲜酱油
  5. 等待蒸好出锅

排骨焖饭

食材
  1. 排骨
  2. 大米
  3. 土豆
  4. 胡萝卜
  5. 生姜
  6. 玉米油
  7. 盐
  8. 六月鲜酱油
做法
  1. 排骨洗干净加水放电饭锅里焯一下
  2. 焯好的排骨拿出来放入电饭锅，加水、油、盐、生姜片，煲排骨汤
  3. 将土豆切块、胡萝卜切丁
  4. 待排骨汤煲好后，将生姜片取出，并加入适量大米，把土豆块、胡萝卜丁放入，加适量六月鲜酱油，煮饭
  5. 待饭煮好后，就大功告成啦

参数校验

发表于 2020-01-02 | 更新于: 2020-12-28 | 分类于 代码
字数统计: 1.2k | 阅读时长 ≈ 4

通用型

资源权限校验

（Alibaba）隶属于用户个人的页面或者功能必须进行权限控制校验

风险：泄露个人隐私信息，或数据被他人恶意修改

比如用户确认某项交易，需要验证所属性；订单系统中，用户确认订单必须要在已收货之后（流程性）

唯一性校验

风险：如果不注意去重，可能会导致下游数据重复或出现对某条数据的多次操作

比如某张用户角色表存在重复数据，需要对某类用户进行扣款操作，不进行校验可能会导致多次扣款的操作

范围参数校验

风险：可能出现数组越界的情况，业务逻辑出错等

分页时，需要检查页数是否在范围内；起始日期不能晚于结束日期、非正、非负判断等

格式化校验

比如BigDecimal传入为非数字字符串时，会报错：NumberFormatException

幂等性校验

某个函数或者接口使用相同参数调用一次或者无限次，其造成的后果是一样的

风险：对于未做校验的短时间内是否重复调用问题，可能导致错误的结果，如重复下单，重复支付等操作

附：Http请求的method中哪些具有幂等性？

GET/PUT/POST/DELETE

Put:因为它直接把实体部分的数据替换到服务器的资源，我们多次调用它，只会产生一次影响，但是有相同结果的 HTTP 方法，所以满足幂等性。

思考：如何解决幂等性校验

用户提交表单校验

用户提交的内容是否包含脚本代码、是否可能存在xss/csrf攻击等

上传文件校验

对于用户上传的文件，需要检查文件大小、文件类型等

Hibernate-validator中的校验

调用第三方接口

调用失败校验

风险：导致业务数据出错

调用第三方接口时，出现异常，要根据业务情况进行处理，比如抛异常、降级等

比如获取分页数据中，因为调用失败没有获取到总量，而设置分页totalCount为0；事实上，数量并不为0

字段判空校验

风险：获取不到数据导致空指针异常

（Alibaba）远程调用返回对象时，一律要求进行空指针判断，防止NPE

需要对各个字段进行校验，因为可能第三方返回的数据结构发生变化，会出现获取为空的情况。例如，返回的json中原先包含id字段，后来没有了。

层级校验

风险：可能导致获取不到某个返回字段，而出现空指针异常的问题

比如微信支付的api中，要求接收到返回内容后，先判断协议字段返回，再判断业务返回，最后判断交易状态

有些数据在return_code返回为Success时，才有返回；有些数据在result_code返回为success时，才有返回；因此要一步一步地校验以获取数据。

特殊值处理

风险：获取到空的、或不符合查询要求的数据

某个字段的格式是否正确，是否包含非法字符，并进行业务需求处理

比如，用户id是否都是有效的，是否出现负数等特殊值

对外提供的接口

（Alibaba）对外提供的开放接口，不管是 RPC/API/HTTP 接口，都需要进行参数校验

问题：写开放的API接口时如何保证数据的安全性？

需要考虑的点如下：

  1. 请求来源(身份)是否合法？
  2. 请求参数被篡改？
  3. 请求的唯一性(不可复制)
直接请求

http://api.test.com/getUsers?param1=value1&param2=value2

请求认证与参数防篡改性

给app分配对应的key、secret，结合参数和secret进行Sign签名，调用API时需要对请求参数进行签名验证

http://api.test.com/getUsers?key=app_key&sign=xxx&param1=value1&param2=value2

按照请求参数名称将所有请求参数按照字母先后顺序排序得到：keyvaluekeyvalue…keyvalue 字符串如：将arong=1,mrong=2,crong=3排序为：arong=1,crong=3,mrong=2。 然后将参数名和参数值进行拼接得到参数字符串：arong1crong3mrong2，将secret加在参数字符串的头部后进行MD5加密，加密后的字符串需大写。即得到签名sign。

请求的唯一性

为了防止重放攻击（Replay-Attack），（最佳实践）加入nonce和timestamp

http://api.test.com/getUsers?**key**=app_key&**sign**=xxx &timestamp=201908281808&nonce=xxx&param1=value1&param2=value2

示例可以参考微信支付相关接口

设计模式相关

抽象工厂模式：抽象工厂、抽象产品，具体工厂创建具体产品的实例。达到实例化不同产品的目的

单例模式：一个类只有一个实例，关键要定义一个私有的构造函数，一个私有的实例，一个公有的获取实例的方法

适配器模式：解决接口不兼容的问题，类似于电源适配器

外观模式：通过重新组合类或程序单元，对外提供统一的接口/界面，要联想到Controller里面的各种Dao或Service，就是这种思想！！

代理模式：让代理去做一些事情，相当于一个中介的角色，处于客户和目标对象之间

观察者模式：建立一种对象之间的依赖关系，当被观察者发生变化，会通知所有的观察者。比如创建订单后，需要继续调用支付接口、通知接口等。主题（subject）与观察者。订单接口作为主题类，支付和通知接口作为观察者，并注册到主题中，只需要调用创建订单接口，并调用notifyObservers方法后，观察者中的update方法就会被执行到。

策略模式：根据环境或条件的不同，选择不同的策略。
