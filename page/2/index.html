http的请求的一些细节

发表于 2019-07-30 | 更新于: 2019-12-23 | 分类于 代码
字数统计: 441 | 阅读时长 ≈ 1
字段 示例 详解
Request URL xxx 请求的URL
Request Method GET 请求的方法
Status Code 200 状态码
Remote Address xxx 远程地址（服务器地址）
Referrer Policy no-referrer-when-downgrade Referrer策略，可参考博客
- Response Headers
Content-Type application/json;charset=UTF-8 响应的内容类型以及编码
Date Tue, 30 Jul 2019 10:30:54 GMT 响应的时间
Transfer-Encoding chunked 传输编码形式，chunked表示数据以一系列分块的形式进行发送
- Request Headers
Host xxx 服务器的域名
Connection keep-alive 决定当前的事务完成后，是否会关闭网络连接。keep-alive表示网络连接是持久的，不会关闭，对同一个服务器的请求可以继续在该连接上完成。
Pragma no-cache
Cache-Control no-cache 缓存机制
X-Requested-With XMLHttpRequest XMLHttpRequest表示为Ajax请求（异步）
User-Agent xxx 识别发起请求的用户代理软件的应用类型、操作系统、软件开发商以及版本号
Content-Type application/json 用于指示资源的MIME类型
Accept */* 告知客户端可以处理的内容类型
Referer xxx 当前请求页面的来源页面的地址
Accept-Encoding gzip, deflate 客户端能够理解的内容编码方式
Accept-Language zh-CN,zh;q=0.9,fr;q=0.8,zh-TW;q=0.7,en;q=0.6 允许客户端声明它可以理解的自然语言
Cookie xxx 服务器发送到用户浏览器并保存在本地的一小块数据，它会在浏览器下次向同一服务器再发起请求时被携带并发送到服务器上。

Access-Control-Allow-Origin: 对于不需具备凭证（credentials）的请求，服务器会以*作为通配符，从而允许所有域都具有访问资源的权限。

设计模式相关

抽象工厂模式：抽象工厂、抽象产品，具体工厂创建具体产品的实例。达到实例化不同产品的目的

单例模式：一个类只有一个实例，关键要定义一个私有的构造函数，一个私有的实例，一个公有的获取实例的方法

适配器模式：解决接口不兼容的问题，类似于电源适配器

外观模式：通过重新组合类或程序单元，对外提供统一的接口/界面，要联想到Controller里面的各种Dao或Service，就是这种思想！！

代理模式：让代理去做一些事情，相当于一个中介的角色，处于客户和目标对象之间

观察者模式：建立一种对象之间的依赖关系，当被观察者发生变化，会通知所有的观察者

策略模式：根据环境或条件的不同，选择不同的策略

RPC在使用形式上像调用本地函数（或方法）一样去调用远程的函数（或方法）。

http接口是在接口不多、系统与系统交互较少的情况下，解决信息孤岛初期常使用的一种通信手段；优点就是简单、直接、开发方便。利用现成的http协议进行传输。但是如果是一个大型的网站，内部子系统较多、接口非常多的情况下，RPC框架的好处就显示出来了

对于dubbo的应用，消费方需要引入提供方的maven包，但是包里面只有接口（只暴露接口），没有具体的实现。所以有rpc在里面，即需要调用远程提供方的方法。

正常情况下，会有多台服务器提供同一个dubbo服务，请求过来，会进行负载均衡；只要保证线上只有一个数据库就可以了。

zookeeper的作用包含：配置管理，名字服务，分布式锁和集群管理。可以把zookeeper理解为一个动物园管理者，负责管理动物园里面的所有动物。每个动物看做一台机器，它们有自己的特点（即配置），有自己的名字（即IP地址等）,因为有很多动物（看做集群），它们在动物园里面活动不能产生冲突（分布式锁）。

nio中最常使用的是多路复用，它是非阻塞的。最主要的有3个组件

  1. Selector（多路复用选择器，类似于netty里面的EventLoop，可以看做是一个线程）
  2. Channel（封装了Socket），注意一个Channel只能注册到一个Selector上，一个Selector可以被多个Channel注册，并监听这些Channel。
  3. ByteBuffer（用于存取数据）

可以类比为一个快递站。一开始快递站的主管一个人负责所有的快递签收，当一卡车的货运过来的时候，它就会把这些快递全部收取。有客户来取得时候，他就得在所有的快递里面一个一个找。

接着，主管想到了一个好的方法。将不同的快递分门别类地摆放，比如申通放在一个架子上，圆通放一个架子，顺丰放一个等等。这时客户来取得时候，他只需要去相应的架子上去找。

后来，主管发现，这样一来，当客户多的时候，他忙不过来，于是，他雇佣了3个员工。这3个人分别负责不同的快递。A负责圆通、申通、中通，B负责顺丰，C负责韵达。这时就好多了。

可以看到，每一件快递可以看做是字节流。而货车可以看作ByteBuffer。每一条快递通道看作Channel，ABC就是Selector。
