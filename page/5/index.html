Java中的单例模式

Posted on 2018-08-17 | In 代码

参考博客：http://cantellow.iteye.com/blog/838473

单例模式的优缺点及使用场景：http://www.cnblogs.com/damsoft/p/6105122.html

优点：

  1. 避免对共享资源的多重占用
  2. 节约系统资源
  3. 对唯一实例的受控访问
  4. 可以控制实例的数目

缺点：

  1. 职责过重，违反了“单一职责原则”
  2. 单例模式中没有抽象层，不方便扩展
  3. 可能会导致内存溢出
  4. 周期过长，可能被垃圾回收

一般采用饿汉式，若对资源十分在意可以采用静态内部类，不建议采用懒汉式及双重检测

spring bean的生命周期，这是spring的基础，一定要熟悉。

  1. 首先得初始化，也就是new的过程。通过构造方法或是工厂方法
  2. 设置对象的属性，setter这些，算是依赖注入。
  3. 检查aware相关的接口，并设置相关依赖
  4. 把bean实例传递给后置处理器的postProcessorBeforeInitialization方法
  5. 调用initial-method初始化方法
  6. 把bean实例传递给后置处理器的postProcessorAfterInitialization方法
  7. 使用Bean
  8. IoC容器关闭之前，调用bean的destroy方法

aop的底层原理

  1. 基于JDK的动态代理，只针对实现了接口的业务对象。
  2. CGLib，通过动态地对目标对象进行子类化来实现aop代理。
  3. 业务对象没有实现任何接口的时候默认使用

Spring Bean的scope，这个scope即它的作用域。在Spring中，一共有5种作用域，分别为：

  1. Singleton，单例，这是默认的作用域类型，表示在IoC容器中只会创建一个共享的Bean，即唯一的实例。
  2. Prototype，每次请求，比如将其注入到另外一个Bean，或是在程序中调用getBean的方法时，都会产生一个新的实例。
  3. Request，每一次Http Request都会实例化一个Bean，只在当前的Http Request中有效。
  4. Session，每一次Http Request都会实例化一个Bean，只在当前的Http Session中有效。
  5. Global Session，只在基于portlet的web应用中才有意义，类似于Session的作用域。

Java抽象类和接口的区别

Posted on 2018-08-17 | In 代码

两者的共同点：

  1. 都可以实现抽象化，都不能实例化
  2. interface的实现类以及abstract class的子类都需要实现已经声明的抽象方法

区别：

  1. 一个类可以实现多个interface，而只能继承一个抽象类
  2. interface强调特定功能的实现，而abstract class强调所属关系，所以interface设计思想是like-a，而abstract class的设计思想是is-a
  3. interface不能有自己的私有变量和方法，而abstract class可以有
  4. interface里面的所有方法都是抽象方法，实现类必须实现，而abstract class的子类可以有选择地实现。具体是，abstract class中的类不都是抽象方法，抽象方法子类必须实现，而非抽象方法，它自己实现。而且对于抽象方法，子类可以选择继续让它抽象，由该子类的子类去实现，这是该子类必须也声明为abstract class。

Java虚拟机

Posted on 2018-08-17 | In 代码

java内存模型：
java虚拟机是一个虚拟的环境，占据着一部分主机的内存。也就是说这是一个中间桥梁，java在运行时不是在赤裸裸的内存上进行的，而是在虚拟机上。

这个区域主要分为2大类。一个是线程隔离的，一个是线程共享的。而它们的生命周期也与此有关。一类是随着虚拟机进程的，而另一类是随着线程的。也就是说线程没了，它也就没了。

一共有5个区域，其实记忆的时候可以类比C/C++中的堆和栈。好！那么这5个区域分别为：

  1. 程序计数区（线程隔离）
  2. Java虚拟机栈（栈，线程隔离）
  3. 本地方法栈（栈，线程隔离）
  4. Java堆（堆，线程共享）
  5. 方法区（堆，线程共享）

但凡涉及到内存，首先就要想到堆和栈。毕竟从本科的时候起，老师们就开始一直灌输这些了。什么栈溢出，堆溢出。其实，说白了，就是递归的深度超过了一定的数目，和内存不够分配了。一个说的是纵向，一个是横线。类比于算法中的广度优先与深度优先，深度优先搜索使用的一定是栈，因为涉及到递归；而广度优先搜索使用到的是队列，它涉及到堆。

仔细思考一下，在C/C++中，是否记得一般的运行时局部变量是存放在栈中的，而程序员申请的空间是存放在堆中的。堆位于地址较高的地方，而栈位于较低的地方。一样的啊！！！Java也是这个套路，虚拟机栈和本地方法栈就是为此服务的，只不过一个给Java方法服务，而另外一个给本地方法（Native）服务！

其实宏观地可以结合算法和数据结构以及计算机体系来理解这些，如下：

计算机体系 算法 数据结构 计算机思想
栈区（内存中连续） 深度优先搜索 栈（stack） 递归
堆区（内存中非连续） 广度优先搜索 堆（heap），优先队列（基于堆） —-

看到了吧，其实并没有想象中的那么难以记忆，关键在于总结和联想！！

再来看看垃圾回收机制吧！

垃圾回收说白了就是针对堆的，仔细想想，java中所有的对象都是存放在堆中的，所以这个地方是最需要进行垃圾回收的。这个时候就要问自己几个问题，第一，什么时候要回收？第二，怎么回收？
关于垃圾回收的方法主要有3种：

  1. 标记-清理（先标记，然后清理）缺点是容易造成碎片
  2. 复制（分为2个区域，每次把存活的对象移动到另外一个区，然后一次性清除当前区的对象）
  3. 标记-整理（先标记，然后整理，也就是说把存活的移动到区域的某一端，然后清理掉端外面的即可）

目前最流行的当然是综合的方法，新生代中的对象几乎都很短命，复制成本低，所以用方法2；而老年代中放的是长命的对象，往往比较大，所以用方法1或3。

好了，有了垃圾回收的方法，那么就要问了，对象放在上面地方呢？有以下3个策略。

  1. 对象优先放在Eden区
  2. 大对象放在老年代
  3. 长期存活的对象将进入老年代（注意这里有个对象年龄的概念，年龄达到一定程度即可进入老年代）

Git的学习

git的工作区就是保存在电脑上的实体文件夹，而stage称为暂存区，add时添加到里面，master就是主分支，而HEAD可以理解为一个指针，指向当前的版本。

先手动删除文件，然后使用git rm <file>和git add <file>效果是一样的。

切换到原来的分支后，文件的内容并没有变化，因为我们是在另外的一个分支上开发的。

Git鼓励大量使用分支：

查看分支：git branch

创建分支：git branch <name>

切换分支：git checkout <name>

创建+切换分支：git checkout -b <name>

合并某分支到当前分支：git merge <name>

删除分支：git branch -d <name>

合并分支时，加上--no-ff参数就可以用普通模式合并，合并后的历史有分支，能看出来曾经做过合并，而fast forward合并就看不出来曾经做过合并。

在本地新建一个分支，并在远程创建一个一样的分支。

在实际开发中，我们应该按照几个基本原则进行分支管理：

首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；我们每个人都有自己的分支，时不时地往dev分支上合并就可以了。

所以，团队合作的分支看起来就像这样：

当修改完bug分支后，要及时地在master上删除该分支，如果切换到dev分支上再删除bug分支，就会报错。

通常涉及到多个人开发，比如A和B。B从github上clone了项目。并且新建并切换到dev分支了。

这个时候A（你）和B（你的小伙伴）对同一个文件进行了修改，发生了冲突。假设B先push了，而你后面想push，就会发生错误。所以可以先pull一下。pull也可能失败，原因是没有指定本地dev分支与远程origin/dev分支的链接，根据提示，设置dev和origin/dev的链接：
git branch --set-upstream-to=origin/dev dev

这回pull成功，但是合并有冲突，需要手动解决，解决的方法和分支管理中的解决冲突完全一样。解决后，add，ci,再push。这样就完美解决了冲突。

这时B如果要再次编辑，首先需要pull一下。然后再进行后续操作。如果他不pull，push后有冲突，那么解决方法同上A所做的。
