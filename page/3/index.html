树遍历的非递归方式

发表于 2018-09-10 | 分类于 代码
字数统计: 969 | 阅读时长 ≈ 3

对于先序遍历：

  1. 当栈或是当前节点不为空，则进行下面的循环
  2. 如果当前节点不会空，则访问该节点，并入栈，并将当前节点指向它的左节点，一直循环，直到当前节点为空
  3. 此时说明走到了最左边，则获取栈顶节点，并弹出它，将当前节点设置为它的右节点

对于中序遍历：
和先序遍历很像，唯一的区别是遍历左节点的时候不访问，而是在后面访问。具体可以这样思考：如果为先序遍历，那么遍历到某个节点时就立即访问；而中序遍历，由于始终要先访问当前节点的左节点，所以就只是先入栈。

对于后序遍历：
由于要确保先访问左右节点，才能访问当前节点，所以要知道左右节点是否已经访问了，因此需要加一个prev，表示上一次访问的节点。有2种思路：
第一种思路：

  1. 将根节点放入栈中，然后当栈不为空就一直循环
  2. 首先获得栈顶节点作为当前节点，如果它的左右节点都为空，或者左右节点不都为空，但是它们之一刚刚被访问过了（即prev为其中之一），那么就可以保证了左右节点已经被访问过，这时可以访问当前节点，更新prev，并出栈
  3. 否则不能访问当前节点（即左右不都为空，且prev为空或者prev不等于其中任意一个），要先访问左右节点，这时把左右节点按照先右后左的顺序入栈，这样访问时就是先左后右了

第二种思路：

  1. 当栈或是当前节点不为空，则进行下面的循环
  2. 如果当前节点不会空，则并入栈，并将当前节点指向它的左节点，一直循环，直到当前节点为空
  3. 这时栈顶节点没有左节点了，这时就看它有没有右节点，如果有的话，并且还没有访问，则设置当前节点为栈顶节点的右节点
  4. 否则，说明右节点不存在或者已经访问过了，这时就可以访问该栈顶节点了，并更新prev和出栈

层次遍历：最简单了

  1. 把根节点放入一个队列
  2. 当队列不为空，就循环下面的步骤
  3. 首先获得队首节点，访问并出队列
  4. 接着把队首节点的左右孩子依次放入队列的尾部，如果存在的话

  4. 对于每个邻居节点，如果它没有被访问过，则入栈，并访问它，设置已经访问过，并立即退出循环（否则就没有深度往下走了）

pdca循环

发表于 2018-09-10 | 分类于 随想
字数统计: 656 | 阅读时长 ≈ 2

任何事情都可以分为pdca四个环节，并且螺旋上升不断解决问题并完成任务。

参考网站：https://zh.wikipedia.org/wiki/PDCA

也就是计划->执行->检查->（act/adjust）调整，其中的调整可以理解为进行沟通等等。

计划：做规划，并分析方案的可行性、分析所需要的时间等

执行：具体的实践、完成任务

检查：可以看作是保证质量的一种行为

调整：通过沟通或是合作来分析任务完成的情况、任务中的需要改进的地方等等

要形成好的闭环思维，而不是恶性的闭环思维。比如熬夜这样的行为就是恶性的

“闭环”的理论根据是“PDCA循环”，由美国质量管理专家休哈特博士提出，被广泛用于现代企业管理中。

“PDCA循环”将管理分为四个阶段：

计划（Plan）

执行（Do）

检查（Check）

行动（Act）

这四个过程不是运行一次就结束，而是周而复始地进行。

Check（检查）、Communicate（沟通）、Clean （清理）、Control（控制）。

时刻让任务涉及的双方知道各自手中任务的完成状况，这不仅是对彼此的督促，对于任务本身来讲，一次沟通就是一次检查点（checkpoint）。

曾在朋友圈看到HR吐槽，最让人“恨得牙根直痒痒”的，是悄无声息把事情搞砸了的人。

企业管理中，存在一种5W2H分析法，也叫七问分析法。

what, why, who, where, when, how, how much(耗费成本)

在开始一项任务前，最重要的是搞清任务的关键以及需求，从而明确什么可以优先，会遇到什么问题，有哪些事项一定要注意……先形成一轮自我可控的闭环思考。

有始有终的闭环，是生活的一份智慧。

我们每个人都生存在很多恶性闭环中，比如由拖延导致的熬夜。

拖延症 → 无法完成任务 → 不得不熬夜 → 第二天精神不济 → 无法完成任务 → 继续熬夜……

这是因为缺乏正向闭环能力中，最核心的一条：建立检查点，对上一个阶段好坏进行总结，把控改进方向。

睡眠就是一个检查点，需要正向的反馈。发现因为熬夜精力不佳，任务被搁置，那就在熬夜这一环进行总结改变。精力充沛，第二天才有可能弥补没有做完的工作，从而终止拖延。

OAuth2是一个关于授权的开放网络标准。

里面的角色主要包括：

  1. 资源所有者（用户）
  2. 用户代理（可以看作是浏览器）
  3. 第三方app（客户端）
  4. Http服务提供商
  5. 认证服务器
  6. 资源服务器，它与认证服务器可以是一台机器或是不同的机器

OAuth2在客户端与服务提供商之间加了一个授权层。

具体步骤如下：

  1. 客户端向用户发送一个认证请求
  2. 用户给客户端授权
  3. 客户端凭借上一步获得的授权，向认证服务器申请令牌
  4. 认证服务器对客户端认证确认无误后，返回给客户端Access Token（令牌）
  5. 客户端凭借令牌向资源服务器申请访问资源
  6. 资源服务器将受保护的资源发送给客户端

最关键的步骤是B，即用户如何给客户端授权，最常用的是授权码模式。

具体步骤如下：

  1. 用户访问客户端，客户端将用户导向认证服务器
  2. 用户选择是否给客户端授权
  3. 假设用户给予了授权，认证服务器将用户导向客户端事先设置好的重定向URI，并附上一个授权码
  4. 客户端收到授权码，附上早先的”重定向URI”，向认证服务器申请令牌，这一步是在客户端的后台完成的，用户无法感知到
  5. 认证服务器核对了授权码和重定向URI，确认无误后，向客户端返回访问令牌

WebApi的安全性

发表于 2018-09-08 | 分类于 代码
字数统计: 690 | 阅读时长 ≈ 2

客户端调用：http://api.xxx.com/getInfo?Id=value，这种方式简单粗暴，在浏览器直接输入http://api.xxx.com/getInfo?Id=value，即可获取到相关的信息了，但是这样的方式会存在很严重的安全性问题，没有进行任何的验证，大家都可以通过这个方法获取到产品列表，导致信息泄露。

  1. 如何验证调用者身份呢？
  2. 如何防止参数被篡改呢？
  3. 如何保证请求的唯一性？
  4. 如何保证请求的唯一性，防止请求被恶意攻击呢？

使用Token+签名认证（微信支付中有这种思想在里面）

  1. 做一个认证服务，提供一个认证的WebApi，用户先访问它获取对应的Token
  2. 用户拿着相应的Token以及请求的参数和服务器端提供的签名算法计算出签名后再去访问指定的Api
  3. 服务器端每次接收到请求就获取对应用户的Token和请求参数，服务器端再次计算签名和客户端签名做对比，如果验证通过则正常访问相应的Api，验证失败则返回具体的失败信息。

使用签名（签名算法+签名参数）

  1. 给每个http请求添加一个签名，服务端来验证签名的合法性，如果签名合法则执行响应的操作，如果签名非法则直接拒绝请求。
  2. 签名算法一般都使用Hash散列算法，常用的有MD5，SHA系列算法。这些算法可以根据不同的输入，计算出不同的结果，而且碰撞的概率很低。
  3. 可以用Http请求的queryString然后加上时间戳还有随机数来作为签名的参数。

WebApi跨域问题解决方案：CORS。什么是跨域问题？

  1. 同源策略：出于安全考虑，浏览器会限制脚本中发起的跨站请求，浏览器要求JavaScript或Cookie只能访问同域下的内容。跨域问题的根本原因是浏览器的安全限制，默认禁止跨域的动态资源请求。
  2. 正是由于这个原因，我们不同项目之间的调用就会被浏览器阻止。比如我们最常见的场景：WebApi作为数据服务层，它是一个单独的项目，我们的MVC项目作为Web的显示层，这个时候我们的MVC里面就需要调用WebApi里面的接口取数据展现在页面上。因为我们的WebApi和MVC是两个不同的项目，所以运行起来之后就存在上面说的跨域的问题。

Synchronized和Lock的区别

发表于 2018-09-07 | 分类于 代码
字数统计: 377 | 阅读时长 ≈ 1

Synchronized的底层实现是一个monitor锁（监视器锁）。每一个对象都有一个监视器锁。主要方法为monitorenter/monitorexit，每个monitor都有一个进入数的概念，当进入数为0时，线程可以获得该对象的锁。当有其它线程占用了monitor，该线程只能阻塞直到进入数为0，才可以重新尝试获取monitor的所有权。

synchronized和lock的区别：

  1. 前者依赖于JVM，属于关键字，而Lock（一般使用reentrantlock，多个线程必须要使用一个reentrantlock才会生效）是基于Java语言的，属于并发包里面的一个类。
  2. 前者是悲观锁（独占锁），其它线程会阻塞直到有个线程释放锁。而后者是乐观锁，它主要内容使用的是CAS操作。同时可以在等待的过程中中断。
  3. 前者无法获取锁的状态，而后者可以。
  4. 前者是独占锁，而后者可以是独占，也可以是共享的（比如读写锁中的读锁）
  5. 前者是非公平锁，后者默认也是非公平锁，但是可以设置成公平的。

关于AQS：https://mubu.com/edit/1FdAd7oAEG

AQS是Lock的核心，叫做抽象队列同步器，里面维持了一个线程阻塞队列存放阻塞了的线程，有一个state变量，用于同步线程之间的共享状态，通过volatile和CAS保证其原子性和可见性。

Java中锁的分类

发表于 2018-09-07 | 分类于 代码
字数统计: 949 | 阅读时长 ≈ 3

公平锁、非公平锁

  1. 公平锁：按照申请锁的顺序来获取锁
  2. 非公平锁：多个线程获取锁的顺序并不是按照申请锁的顺序，有可能后申请的线程比先申请的线程优先获取锁。有可能，会造成优先级反转或者饥饿现象
  3. ReentrantLock，可以是公平的，也可以是非公平的。默认为非公平。Synchronized，非公平锁

可重入锁

  1. 又名递归锁，指在同一个线程在外层方法获取锁的时候，在进入内层方法会自动获取锁

独享锁、共享锁

  1. 独享锁：该锁一次只能被一个线程所持有
  2. 共享锁：该锁可以被多个线程所持有
  3. ReentrantLock，为独享锁。但是ReadWriteLock，其读锁是共享锁，而写锁是独享锁。读锁的共享锁保证了并发读是非常高效的；读写，写写的过程是互斥的
  4. Synchronized，为独享锁

互斥锁、读写锁

  1. 上面讲的独享锁、共享锁是一种广义的说法，互斥锁、读写锁是具体的实现
  2. 互斥锁在Java中的具体实现：ReentrantLock
  3. 读写锁在Java中的具体实现：ReadWriteLock

乐观锁、悲观锁

  1. 不是具体的锁，而是指看待并发同步的角度
  2. 悲观锁认为，不加锁的并发操作一定会出现问题
  3. 乐观锁认为，不加锁的并发操作是没有事情的
  4. 悲观锁适合写操作非常多的场景，乐观锁适合读操作非常多的场景，不加锁会带来大量的性能提升
  5. 悲观锁在Java中的使用，就是利用各种锁
  6. 乐观锁在Java中的使用，是无锁编程，常常采用的是CAS算法，典型的例子就是原子类，通过CAS自旋实现原子操作的更新

分段锁

  1. 是一种锁的设计，并不是具体的锁
  2. ConcurrentHashMap的并发就是通过分段锁的形式来实现高效的并发操作
  3. ConcurrentHashMap中的分段锁称为Segment，其继承自ReentrantLock，它类似于HashMap的结构，内部拥有一个Entry数组，数组中的每个元素又是一个链表
  4. put的时候，不是对整个hashmap进行加锁，而是先通过hashcode来知道它要放在哪一个分段中，然后对这个分段进行加锁。所以当多线程put的时候，只要不是放在一个分段中，就实现了真正的并行插入
  5. 在统计size的时候，就要获取hashmap的全局信息，这时需要获取所有分段的锁才能统计

偏向锁、轻量级锁、重量级锁（锁的升级）

  1. 锁的状态，针对Synchronized
  2. 偏向锁是指一段同步代码一直被一个线程所访问，那么该线程会自动获取锁。降低获取锁的代价
  3. 轻量级锁是指当锁是偏向锁的时候，被另一个线程所访问，偏向锁就会升级为轻量级锁，其他线程会通过自旋的形式尝试获取锁，不会阻塞，提高性能
  4. 重量级锁是指当锁为轻量级锁的时候，另一个线程虽然是自旋，但自旋不会一直持续下去，当自旋一定次数的时候，还没有获取到锁，就会进入阻塞，该锁膨胀为重量级锁。重量级锁会让其他申请的线程进入阻塞，性能降低

自旋锁（忙等待）

  1. 在Java中，自旋锁是指尝试获取锁的线程不会立即阻塞，而是采用循环的方式去尝试获取锁，这样的好处是减少线程上下文切换的消耗，缺点是循环会消耗CPU

计算机网络

ISO7层网络模型

  1. 应用层：处理网络应用
  2. 表示层：数据表示
  3. 会话层：互联主机通信
  4. 传输层：端到端链接（TCP/UDP协议）
  5. 网络层：寻址和最短路径（IP协议）
  6. 数据链路层：接入介质
  7. 物理层：二进制传输

5层无表示层和会话层

TCP

三次握手

SYN=1表示请求连接

  1. 建立链接时C端发送SYN包到S端（SYN=1，seq=x），C端进入SYS_SENT状态
  2. S端在收到C端的SYN包以后必须确认C端的SYN包，所以发送（SYN=1，ACK=1，seq=y，ack=x+1）到C端，然后S端进入SYS_RCVD状态
  3. C端再收到S端发送的SYN和ACK包以后，向服务器发送ACK包（ACK=1，ack=y+1），然后C端进入ESTABLISHED状态，S端在收到C端的确认包后也进入ESTABLISHED状态
四次挥手
  1. C端向S端发送释放链接请求，发送FIN=1，seq=x，发送完后进入FIN_WAIT_1状态
  2. S端在收到C端的请求后进入CLOSE_WAIT状态，然后发送给C端一个确认包（ACK=1，seq=x+1）。C端这时候不能再发送数据了，只能发送确认报文，C端在接受到S端发送的确认包以后进入FIN_WAIT_2状态
  3. 当S端所有的数据都传完的时候，向C端发送释放链接报文（FIN=1，seq=y），发完之后自己进入LAST_WAIT状态
  4. C端收到S端的请求后，发送一个确认报文，自己进入TIME_WAIT状态，而不是立马关闭，因为C端发送的确认报文可能会丢失，丢失的话S端就会重传一个FIN，也就是释放连接报文，这个时候C端必须还没关闭。当S端接受到确认报文后，就进入CLOSE状态。但是由于上面所说原因，C端必须等待一定的时间才能够进入CLOSE状态。而这个等待的时间长度是2*msl（由于可能出现不可靠的连接，当C端发送确认后，可能S端没有收到，如果它没有收到，它就会再次发FIN，这时C端必须再次发送确认，因此C端不能急着关闭；而要等一段时间，如果这个时间S端没有消息过来，说明S端收到了确认，这时C端才能关闭）

2倍报文最大生存时间，任何报文在网络上存在的最长时间，超过这个时间报文将被丢弃。等待2MSL时间主要目的是怕最后一个ACK包对方没收到，那么对方在超时后将重发第三次握手的FIN包。msl 30秒到2分钟

为什么一个是三次，一个是四次；关键在于关闭连接时，当S端收到FIN后，它只能暂时确认，而不能同时FIN，因为它也许还要继续发数据给C端，因此造成了ACK和FIN不能同时发送过去。这样就多了一个流程。

握手攻击

DoS-拒绝服务攻击
TCP连接时，攻击者故意不完成三次握手，导致资源浪费

解决方法：限制同时打开的SYN半连接数；缩短SYN半连接的TIME_OUT时间；启用防火墙

为什么是三次握手不是两次

两次握手可能会发生死锁的情况。C端发送给S端一个请求，S端在收到后并发送确认（S端认为连接成功）。如果S端发送的确认丢失，C端会认为连接还未成功而忽略S端发送的任何数据。而S端会在发送数据超时后重发同样的分组，进而形成死锁。

UDP

用户数据报协议，无连接，面向报文，无阻塞控制，尽最大努力交付。

TCP与UDP的区别

  1. TCP是面向连接的，可靠的数据传输，而UDP是非面向连接的，不可靠的数据传输
  2. TCP的传输单位是TCP报文段，UDP则是用户数据报
  3. TCP注重安全，UDP传输更快

关于滑动窗口与拥塞控制

  1. 发送方和接受方都持有自己的窗口值
  2. 慢开始：发送发从初始的cwnd=1，慢慢指数级增长
  3. 当达到了ssthresh的值后，开始拥塞算法，即cwnd加法增长，每次+1
  4. 当网络中出现拥塞时，设置ssthresh的值为发送拥塞窗口值的一半，并设置cwnd=1，再次慢开始
  5. 快重传，接收方每次收到失序的报文后立即发出重复确认，发送方连续3次接收到重复确认时，立即重传。配合6使用
  6. 快恢复，在4中，设置cwnd=ssthresh，并使用拥塞算法（加1）增长

mysql中的undo、redo

发表于 2018-09-03 | 分类于 代码
字数统计: 412 | 阅读时长 ≈ 1

mysql中的日志文件分为undo log和redo log。

undo日志记录的是数据修改之前的值，主要保证数据的一致性。因为当修改出现异常，可以用来回滚到之前的状态。

undo日志分为2中类型，一种是insert_undo，记录插入的唯一键值；还有一种是update_undo，记录修改的唯一键值和修改之前的值。

redo日志记录的是数据修改之后的值，主要保证数据的原子性和持久性。在数据库对数据做修改的时候，会把数据页从磁盘读到buffer pool里面，如果这时发生数据库重启，那么这些数据就无法同步到磁盘了。

使用redo日志有2个好处

  1. 在发生数据库重启（或crash）时，buffer pool的数据还没有刷新到磁盘，通过redo日志可以找到需要刷新到磁盘的记录。
  2. buffer pool刷新到磁盘是一个随机IO的过程，效率较差；而把buffer pool的数据记录到redo log是一个顺序IO，可以提高事务提交的速度。

通过undo + redo的设计，使得IO性能得到提高。

  1. 缓存到buffer区，批量写入日志（redo log buffer）
  2. 并发事务共享redo log存储空间，所以其它未提交的事务日志也会被提交
  3. redo log只是顺序追加

Mysql的InnoDB存储引擎使用的恢复策略：进行恢复时，重做所有的事务，包括未提交的和回滚的事务，然后再根据undo log回滚那些未提交的事务。
