树遍历的非递归方式

Posted on 2018-09-10 | In coding

对于先序遍历：

  1. 当栈或是当前节点不为空，则进行下面的循环
  2. 如果当前节点不会空，则访问该节点，并入栈，并将当前节点指向它的左节点，一直循环，直到当前节点为空
  3. 此时说明走到了最左边，则获取栈顶节点，并弹出它，将当前节点设置为它的右节点

对于中序遍历：
和先序遍历很像，唯一的区别是遍历左节点的时候不访问，而是在后面访问。具体可以这样思考：如果为先序遍历，那么遍历到某个节点时就立即访问；而中序遍历，由于始终要先访问当前节点的左节点，所以就只是先入栈。

对于后序遍历：
由于要确保先访问左右节点，才能访问当前节点，所以要知道左右节点是否已经访问了，因此需要加一个prev，表示上一次访问的节点。有2种思路：
第一种思路：

  1. 将根节点放入栈中，然后当栈不为空就一直循环
  2. 首先获得栈顶节点作为当前节点，如果它的左右节点都为空，或者左右节点不都为空，但是它们之一刚刚被访问过了（即prev为其中之一），那么就可以保证了左右节点已经被访问过，这时可以访问当前节点，更新prev，并出栈
  3. 否则不能访问当前节点（即左右不都为空，且prev为空或者prev不等于其中任意一个），要先访问左右节点，这时把左右节点按照先右后左的顺序入栈，这样访问时就是先左后右了

第二种思路：

  1. 当栈或是当前节点不为空，则进行下面的循环
  2. 如果当前节点不会空，则并入栈，并将当前节点指向它的左节点，一直循环，直到当前节点为空
  3. 这时栈顶节点没有左节点了，这时就看它有没有右节点，如果有的话，并且还没有访问，则设置当前节点为栈顶节点的右节点
  4. 否则，说明右节点不存在或者已经访问过了，这时就可以访问该栈顶节点了，并更新prev和出栈

层次遍历：最简单了

  1. 把根节点放入一个队列
  2. 当队列不为空，就循环下面的步骤
  3. 首先获得队首节点，访问并出队列
  4. 接着把队首节点的左右孩子依次放入队列的尾部，如果存在的话

WebApi的安全性

Posted on 2018-09-08 | In coding

客户端调用：http://api.xxx.com/getInfo?Id=value，这种方式简单粗暴，在浏览器直接输入http://api.xxx.com/getInfo?Id=value，即可获取到相关的信息了，但是这样的方式会存在很严重的安全性问题，没有进行任何的验证，大家都可以通过这个方法获取到产品列表，导致信息泄露。

  1. 如何验证调用者身份呢？
  2. 如何防止参数被篡改呢？
  3. 如何保证请求的唯一性？
  4. 如何保证请求的唯一性，防止请求被恶意攻击呢？

使用Token+签名认证（微信支付中有这种思想在里面）

  1. 做一个认证服务，提供一个认证的WebApi，用户先访问它获取对应的Token
  2. 用户拿着相应的Token以及请求的参数和服务器端提供的签名算法计算出签名后再去访问指定的Api
  3. 服务器端每次接收到请求就获取对应用户的Token和请求参数，服务器端再次计算签名和客户端签名做对比，如果验证通过则正常访问相应的Api，验证失败则返回具体的失败信息。

使用签名（签名算法+签名参数）

  1. 给每个http请求添加一个签名，服务端来验证签名的合法性，如果签名合法则执行响应的操作，如果签名非法则直接拒绝请求。
  2. 签名算法一般都使用Hash散列算法，常用的有MD5，SHA系列算法。这些算法可以根据不同的输入，计算出不同的结果，而且碰撞的概率很低。
  3. 可以用Http请求的queryString然后加上时间戳还有随机数来作为签名的参数。

WebApi跨域问题解决方案：CORS。什么是跨域问题？

  1. 同源策略：出于安全考虑，浏览器会限制脚本中发起的跨站请求，浏览器要求JavaScript或Cookie只能访问同域下的内容。跨域问题的根本原因是浏览器的安全限制，默认禁止跨域的动态资源请求。
  2. 正是由于这个原因，我们不同项目之间的调用就会被浏览器阻止。比如我们最常见的场景：WebApi作为数据服务层，它是一个单独的项目，我们的MVC项目作为Web的显示层，这个时候我们的MVC里面就需要调用WebApi里面的接口取数据展现在页面上。因为我们的WebApi和MVC是两个不同的项目，所以运行起来之后就存在上面说的跨域的问题。

Redis

Posted on 2018-09-03 | In coding

redis的list的底层实现是一个双向链表，也可以使用它实现一个简单的消息队列，生产者push，消费者pop。

hash，类似于python里面的dict，C++里面的unordered_map，Java里面的HashMap。底层实现为HashMap，可以用来存储用户的id及其信息（比如以json格式保存）。

set的底层实现是一个value值为null的hashmap，它根据hash的方式快速去重。

sorted set底层实现是hashmap + skiplist。为什么要用2个呢？当进行查询的时候，可以使用hashmap保证其$O(1)$级别的复杂度；而在进行更新操作时，如果只有hashmap，那么需要进行重排，时间复杂度为$O(nlogn)$，而如果加入跳跃表，那么只需要$O(logn)$的复杂度。所以结合使用。

rdb/aof持久化

rdb:数据库状态（所有数据库中的所有键值对数据）和rdb文件相互转存，可以看做是快照。文件格式为二进制文件。

save会阻塞服务器进程，导致其不能处理其它任何命令请求；bgsave会派生出一个子进程去完成，父进程不受影响。可以设置bgsave自动执行的参数（保存条件，当任意一个条件满足，则自动执行），包括时间间隔和修改数。在redis服务器中则保存了上一次保存的时间戳lastsave和距离上次保存总计的修改次数dirty计数器。

aof:通过保存所有修改数据库的写命令请求来记录服务器的数据库状态。

aof文件中的所有命令都是以Redis命令请求协议的格式保存。命令请求会先保存到aof缓冲区里面，之后再定期写入并同步到aof文件中。

Redis的发布/订阅功能

可以用来订阅频道或是模式

  1. 订阅频道使用的dict（字典），key保存频道，而value保存订阅它的客户端信息。
  2. 订阅模式使用的list（链表），list里面保存了pubsubPattern结构，该结构里面有2个属性，一个是pattern，一个是client（只包含一个client）。
  3. publish命令会向频道的所有订阅者发消息，并向所有与该频道匹配的模式的订阅者发消息。
  4. pubsub的3个子命令，channels用来返回当前服务器被订阅的频道；numsub接受多个频道作为参数，并返回这些频道的订阅者数量；numpat返回该服务器所有被订阅模式的数量。

TCP

三次握手

SYN=1表示请求连接

  1. 建立链接时C端发送SYN包到S端（SYN=1，seq=x），C端进入SYS_SENT状态
  2. S端在收到C端的SYN包以后必须确认C端的SYN包，所以发送（SYN=1，ACK=1，seq=y，ack=x+1）到C端，然后S端进入SYS_RCVD状态
  3. C端再收到S端发送的SYN和ACK包以后，向服务器发送ACK包（ACK=1，ack=y+1），然后C端进入ESTABLISHED状态，S端在收到C端的确认包后也进入ESTABLISHED状态
四次挥手
  1. C端向S端发送释放链接请求，发送FIN=1，seq=x，发送完后进入FIN_WAIT_1状态
  2. S端在收到C端的请求后进入CLOSE_WAIT状态，然后发送给C端一个确认包（ACK=1，seq=x+1）。C端这时候不能再发送数据了，只能发送确认报文，C端在接受到S端发送的确认包以后进入FIN_WAIT_2状态
  3. 当S端所有的数据都传完的时候，向C端发送释放链接报文（FIN=1，seq=y），发完之后自己进入LAST_WAIT状态
  4. C端收到S端的请求后，发送一个确认报文，自己进入TIME_WAIT状态，而不是立马关闭，因为C端发送的确认报文可能会丢失，丢失的话S端就会重传一个FIN，也就是释放连接报文，这个时候C端必须还没关闭。当S端接受到确认报文后，就进入CLOSE状态。但是由于上面所说原因，C端必须等待一定的时间才能够进入CLOSE状态。而这个等待的时间长度是2*msl（由于可能出现不可靠的连接，当C端发送确认后，可能S端没有收到，如果它没有收到，它就会再次发FIN，这时C端必须再次发送确认，因此C端不能急着关闭；而要等一段时间，如果这个时间S端没有消息过来，说明S端收到了确认，这时C端才能关闭）

2倍报文最大生存时间，任何报文在网络上存在的最长时间，超过这个时间报文将被丢弃。等待2MSL时间主要目的是怕最后一个ACK包对方没收到，那么对方在超时后将重发第三次握手的FIN包。msl 30秒到2分钟

为什么一个是三次，一个是四次；关键在于关闭连接时，当S端收到FIN后，它只能暂时确认，而不能同时FIN，因为它也许还要继续发数据给C端，因此造成了ACK和FIN不能同时发送过去。这样就多了一个流程。

握手攻击

DoS-拒绝服务攻击
TCP连接时，攻击者故意不完成三次握手，导致资源浪费

解决方法：限制同时打开的SYN半连接数；缩短SYN半连接的TIME_OUT时间；启用防火墙

为什么是三次握手不是两次

两次握手可能会发生死锁的情况。C端发送给S端一个请求，S端在收到后并发送确认（S端认为连接成功）。如果S端发送的确认丢失，C端会认为连接还未成功而忽略S端发送的任何数据。而S端会在发送数据超时后重发同样的分组，进而形成死锁。

UDP

用户数据报协议，无连接，面向报文，无阻塞控制，尽最大努力交付。

TCP与UDP的区别

  1. TCP是面向连接的，可靠的数据传输，而UDP是非面向连接的，不可靠的数据传输
  2. TCP的传输单位是TCP报文段，UDP则是用户数据报
  3. TCP注重安全，UDP传输更快

关于滑动窗口与拥塞控制

  1. 发送方和接受方都持有自己的窗口值
  2. 慢开始：发送发从初始的cwnd=1，慢慢指数级增长
  3. 当达到了ssthresh的值后，开始拥塞算法，即cwnd加法增长，每次+1
  4. 当网络中出现拥塞时，设置ssthresh的值为发送拥塞窗口值的一半，并设置cwnd=1，再次慢开始
  5. 快重传，接收方每次收到失序的报文后立即发出重复确认，发送方连续3次接收到重复确认时，立即重传。配合6使用
  6. 快恢复，在4中，设置cwnd=ssthresh，并使用拥塞算法（加1）增长

mysql中的undo、redo

Posted on 2018-09-03 | In coding

mysql中的日志文件分为undo log和redo log。

undo日志记录的是数据修改之前的值，主要保证数据的一致性。因为当修改出现异常，可以用来回滚到之前的状态。

undo日志分为2中类型，一种是insert_undo，记录插入的唯一键值；还有一种是update_undo，记录修改的唯一键值和修改之前的值。

redo日志记录的是数据修改之后的值，主要保证数据的原子性和持久性。在数据库对数据做修改的时候，会把数据页从磁盘读到buffer pool里面，如果这时发生数据库重启，那么这些数据就无法同步到磁盘了。

使用redo日志有2个好处

  1. 在发生数据库重启（或crash）时，buffer pool的数据还没有刷新到磁盘，通过redo日志可以找到需要刷新到磁盘的记录。
  2. buffer pool刷新到磁盘是一个随机IO的过程，效率较差；而把buffer pool的数据记录到redo log是一个顺序IO，可以提高事务提交的速度。

通过undo + redo的设计，使得IO性能得到提高。

  1. 缓存到buffer区，批量写入日志（redo log buffer）
  2. 并发事务共享redo log存储空间，所以其它未提交的事务日志也会被提交
  3. redo log只是顺序追加

Mysql的InnoDB存储引擎使用的恢复策略：进行恢复时，重做所有的事务，包括未提交的和回滚的事务，然后再根据undo log回滚那些未提交的事务。

数据库事务四种隔离级别

Posted on 2018-09-02 | In coding

未授权读取（read uncommitted）

授权读取（read committed）

可重复读取（repeatable read）

序列化（serializable）

为什么会提出这四个概念，是因为在一个事务执行的过程中，可能出现以下几种情况：

  1. 更新丢失：两个事务同时更新一行数据，一个事务对数据的更新把另外一个事务对数据的更新覆盖了。不能应对并发
  2. 脏读：一个事务读取到了另外一个事务未提交的数据操作结果
  3. 不可重复读：一个事务对同一行数据重复读取2次，但是得到了不同的结果
  4. 虚读：事务T1读取了某一数据后，事务T2对其做了修改，当事务T1再次读该数据时得到了与第一次不同的值
  5. 幻读：事务在操作的过程中进行2次查询，第一次查询到的结果包含了第一次查询中未出现的数据或者缺少了第一次查询中出现的数据。这是因为在两次查询过程中另外一个事务插入数据造成的

未授权读取（我在写，你就不能写，但是你可以读）就是一个事务可以读取另一个未提交事务的数据。
如果一个事务已经开始写数据，则另外一个事务不允许同时进行写操作，但是允许其它事务读此行数据。该隔离级别可以通过“排他写锁”实现。
避免了更新丢失，却可能出现脏读。也就是说事务B读取到了事务A未提交的数据

授权读取（我在读，你可以读；但是我在写，如果还没有提交，你不能读）
就是一个事务要等另一个事务提交后才能读取数据。读取数据的事务允许其它事务继续访问该行数据，但是未提交的写事务将会禁止其它事务访问该行。
避免了脏读，但是可能出现不可重复读。事务A事先读取了数据，事务B紧接着更新了数据，并提交了事务，而事务A再次读取该数据时，数据已经发生了变化。

可重复读取（我在读，你不能写，可以读）
就是在开始读取数据（事务开启）时，不再允许修改操作读取数据的事务将会禁止写事务（但允许读事务），写事务则禁止任何其它的事务。避免了不可重复读和脏读，但是有时可能出现幻读。可以通过“共享读锁”和“排他写锁”实现。
重复读可以解决不可重复读问题。不可重复读对应的是修改，即UPDATE操作。但是可能还会有幻读问题。因为幻读问题对应的是插入INSERT操作，而不是UPDATE操作。

序列化，事务只能一个接着一个地执行，但不能并发执行。可以解决幻读问题

总结：对于大多数应用程序，可以优先考虑把数据库的隔离级别设置为read committed。可以避免脏读，而且有较好的并发性能。尽管可能导致不可重复读、幻读和第二类丢失更新这些并发问题。可以由应用程序采用悲观锁或乐观锁来控制。MySQL默认的隔离级别就是repeatable read。

计算机思维

Posted on 2018-08-26 | In coding

在计算机中，有如下的思维，一定要熟记于心，无论遇到什么问题，都可以往这方面思考，正所谓万精油！！

  1. 缓存思想：缓存思想特别重要，这是很多设计中必出现的，无论在操作系统层面还是具体应用层面。比如在内存和CPU之间有寄存器；比如在Redis中有缓存机制；Mybatis也有缓存机制，当某些记录很大，而且基本上不会变，且查询频率很高，那么利用缓存是明智的选择。消息队列
  2. Hash+链表、树形结构思想：说到hash一般会想到hashtable，hashmap等等，树形结构就更多了。共同点是，说到查找，它们是高频词汇。对于无序的东西，我们的查找时间复杂度是n级别的。所以为了优化，会采用2种措施，一种是建立索引，即hash表，计算key值，然后放到对应的位置，如果出现碰撞，通常是加个链表。这样复杂度可以控制在1级别的。凡事都有优缺点，虽然查询快了，但是它是无序的。而如果采用第二种措施，即用树形结构，比如二叉树（红黑树）、B+树等。基本是可以实现logn级别的，虽然慢一些，但其优点是有序。凡是涉及到查找，要想到它们！在ElasticSearch中有这些思想的存在（倒排索引，基于单词文档矩阵模型）。
  3. 分布式思想：这个不用说了，当应用大了以后，必然出现。涉及到分布式，那么一定会涉及到同步、一致性等等问题。
  4. 解耦思想：这个思想很重要，无论在什么地方都会体现，任何系统，都应当考虑到解耦。
  5. top-down思想：任何时候可以先从宏观整体考虑，然后再到具体的部分。递归是一种体现。
  6. 主从思想：这一思想常常出现在分布式中
  7. DP思想：这个不用说，算法中最难以理解，也是最能解决复杂问题的一种思想。
  8. 双倍扩容思想：对于一些动态容器，比如C++中的vector扩容，java中的hashmap扩容，Redis中的dict的rehash过程。多多少少有基于原来容量2倍来进行扩容的思想。
  9. 拦截过滤思想
  10. 注册通知（发布、通知）思想，反应器模式

IO模型

Posted on 2018-08-18 | In coding

关于5种I/O模型的理解

以喝咖啡举例， 在网上点了咖啡粉外卖，要等外卖小哥送咖啡，然后泡咖啡，喝咖啡

  1. 阻塞式I/O：一直站在家里等咖啡（其它什么事不做）送到，然后泡咖啡
  2. 非阻塞式I/O：不停地给外卖小哥打电话（其它什么事不做），送到后泡咖啡
  3. IO多路复用：一开始和小哥说好，到了后打电话，同时站在家里等（于此同时，他还在等其它的外卖，也是什么事不做），一旦咖啡到了，就泡咖啡
  4. 信号驱动式IO：干自己的事情，外卖小哥到了后打电话让你取咖啡，然后你泡咖啡
  5. 异步IO：一直干自己的事情，整个事情都让女朋友来做，做好了通知自己过去喝

前面3种提到的IO都是同步的，与之相对应的是异步IO。这是一个维度，还有一个是阻塞的维度。传统的是阻塞的IO，不管是C端还是S端，都是阻塞的。接着是非阻塞的IO，主要通过C端的轮询来实现。后面的多路复用，它也是非阻塞的，通过选择器和通道来实现。主要采用的是反应器（Reactor）模式。涉及到事件的注册和订阅。

很早很早之前，Java的api只支持由本地系统套接字库提供的阻塞函数。阻塞体现在服务器端的监听过程，直到有新的连接，这时就建立一个双方通信的socket。接着它继续监听。其缺点是只能同时处理一个连接，要管理多个并发的客户端连接，需要开启多个线程。
